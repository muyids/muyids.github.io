# 索引的基本原理

索引用来快速地寻找那些具有特定值的记录。如果没有索引，一般来说执行查询时遍历整张表。

索引的原理：就是把无序的数据变成有序的查询

  1. 把创建了索引的列的内容进行排序
  2. 对排序结果生成倒排表
  3. 在倒排表内容上拼上数据地址链
  4. 在查询的时候，先拿到倒排表内容，再取出数据地址链，从而拿到具体数据

# mysql聚簇和非聚簇索引的区别

  • 聚簇索引，B+树的叶子节点 包括 索引 + 行数据；
  • 非聚簇索引，叶子节点 只有 索引 + 行数据的地址

InnoDB使用的是聚簇索引，包括一棵主键B+树(主键索引树)和n棵辅助键B+树（辅助键索引树），主键B+树的叶子节点存储了主键和行数据，辅助键B+树存储了辅助键和主键；当通过主键检索数据时，只查找主键B+树，找到对应叶子节点，直接返回数据；通过辅助键检索数据时，需要先查找辅助键B+树，找到对应的主键，然后通过主键检索到相应的叶子节点数据；

MyISAM使用的是非聚簇索引，无论是主键B+树还是辅助键B+树，叶子节点保存的都是行数据的引用地址，而行数据单独进行存储；通过主键检索时，查找主键B+树，得到行数据引用地址，然后找到对应行数据返回；通过辅助键检索时，查找辅助键B+树，得到行数据引用地址，然后找到对应行数据返回。

聚簇索引的优势：

  1. 叶子节点保存了主键和行数据，找到叶子节点就得到了数据，不需要通过引用地址查找数据；通过主键检索，获取数据更快
  2. 辅助键B+树使用主键作为”指针“，而不是使用引用地址，减少了出现行移动或数据页分裂时辅助索引的维护工作；聚簇索引就可以保证不管这个主键B+树的节点如何变化，辅助键B+树都不受影响。

# mysql索引的数据结构，各自优劣

  • B+树索引
  • Hash索引
  • 全文索引

B+树索引

  1. 多路平衡查找树，从根节点到每个叶子节点的高度差值不超过1
  2. 非叶子节点不存储data，只存储索引（冗余）,可以放更多的索引；叶子节点包含所有索引字段
  3. 同层节点间有指针相互链接
  4. 从根节点到叶子节点的搜索效率基本相当，不会出现大幅波动，而且基于索引的顺序扫描时，也可以利用双向指针快速左右移动，效率非常高
  5. B+树索引被广泛应用于数据库、文件系统等场景。

Hash索引

基于Hash表实现；查询时，通过hash函数计算出数据存储位置，从而获得实际数据。

数据库索引为什么用b+树而不用hash表？

  • Hash索引不能实现范围查找

  • Hash索引无法排序

  • hash索引无法根据前缀查询，如like 'aaa%'

  • hash索引无法用于比较

  • hash索引虽然在等值查询上较快，但是不稳定，当某个键值存在大量重复的时候，发生hash碰撞，此时效率极差。

# 索引设计的原则

核心原则：通过索引过滤出尽量少的数据，减少随机IO，增加查询性能

  • 出现在where子句中的列

  • 有外键的数据列（出现在连接子句中指定的列）

  • 基数较小的表，索引效果较差，没有必要在此列建立索引

  • 使用短索引。如果对长字符串列进行索引，应该指定一个前缀长度，这样能够节省大量索引空间，

    如果搜索词超过索引前缀长度，则使用索引排除不匹配的行，然后检查其余行是否可能匹配。

  • 不要过度索引。索引需要额外的磁盘空间，并降低写操作的性能。在修改表内容的时候，索引会进

    行更新甚至重构，索引列越多，这个时间就会越长。所以只保持需要的索引有利于查询即可。

  • 不能有效区分数据的列不适合做索引列(如性别，男女未知，最多也就三种，区分度实在太低)

  • 尽量的扩展索引，不要新建索引。比如表中已经有a的索引，现在要加(a,b)的索引，那么只需要修

    改原来的索引即可。

  • Innodb表必须有个主键，目的：建立主键索引树；（Innodb表的主键索引树生成规则：优先Primary key，其次选择Unique Index Key，如果都没有使用隐藏行id）

# 索引的最左匹配原则

MySQL建立联合索引时会遵守最左前缀匹配原则，即最左优先，在检索数据时从联合索引的最左边开始匹配。
